2017年度总结

发表于 2017-12-18 |

项目总结

基础平台

全年参与

2017/02~2017/03

产出:

  • 消息监控(m3)
  • license的基础功能
  • 基于maven的文档生成法
  • 版本发布1.0.8

年初项目基本的需求功能处于初始版本完成阶段(停留在基础的Action-Logic-Eao阶段), 由于2016~2017年初没有统一的开发人员计划与规范使得沉淀下来的代码显得没有章法. 2017年初后, 按总部要求需要将现阶段的平台研发独出来成立研发中心, 所以把公司内svn下的平台代码转移到研发中心下, 在研发中心中需要进行过度开发(2017/02~2017/03). 公司内部为了减少劳动力大量流入研发中心, 选择将大部分新人员工转入过度开发阶段的劳动(2017/02), 公司虽然为每人都划分了一定的平台模块, 但人员对整体的平台不了解不熟悉, 最终平台的功能研发出现了停滞. 也正是因为新人员对平台以及代码规范的随意, 进一步加剧了代码的混乱.

在这期间对新人员与研发中心接手人员工进行了基础的模块功能释义与基本功能的培训, 完成了对平台基础功能与应用功能的划分. 重整项目结构, 为研发中心还原开发环境与nexu私服的搭建与备份, 并最终在3月底将整体项目转入研发中下.

不足: 没有起到做够的管理功能, 没能将人员约束在统一的开发规范下.

2017/04

产出:

  • module-generator(技术研讨与数据建模)

基础平台的代码移交后不再对研发中心的代码负责, 转而开始对公司内svn上的代码进行重新整理. 公司内待整理的内容其实已经在研发中心中实践了一遍, 所以需要进一步的通过实践找出最优的方案.

  • 在原有的基础上对项目的结构按组件-应用进行目录划分
  • 重新编排GAV
  • 优化maven pom.xml配置
  • 优化文档的生成与管理
  • 配合公司现有jenkins尝试自动化部署

月底(20)接到新的需求, 需要按现在的代码模式来完成一套代码生成的模版, 开始转入需求与技术研究, 最终决定保留现在的ALE(Action-Logic-Eao)模式 + Apache Freemarker来做实现. 而此需求需要建立在对应的数据模型上, 对现在的ALE进行数据建模.

2017/05

产出:

  • module-generator(进入开发)
  • function

对ALE进行数据建模后的代码化, 以及对应的模版编写.

数据模型在高度抽象与便捷使用间进行权衡, 如果是作为一套单独的基础功能, 用于生成代码需要有足够细粒度的模型来满足对生成代码的要求. 权衡后的结果为module-generator的数据模型尽量保持在足够细的粒度(可实现字段-方法-注解等粒度的生成), 但是需要提供完整的Api来对细粒度的数据进行创建, 意图满足需求又不损害基础功能的可扩展性.

尝试spring-webmvc来替换现在的struts2(Action)部分的功能, 所受到的历史代码影响非常大. 如何说服领导进行替换是非常困难的, 恐难成功.

引入srping-webmvc需要对现有代码做兼容(主要兼容的压力来自前端部分的数据提交结构以及连接后台的bean), 需要自定义beanFactory, 前端数据提交的封装.

  • 自定义ejbBeanFactory, 以及替换了org.springframework.context.annotation.AnnotationConfigUtils.COMMON_ANNOTATION_PROCESSOR_BEAN_NAME
  • 基于spring-webmvc扩展的一套HandlerMethodArgumentResolver, HandlerMethodReturnValueHandler来解决数据提交与传递

在5月中引入spring-webmvc阶段大致了解到了现阶段的控制台(console)的页面显示逻辑混乱, 页面切换间的体验差等一系列不大不小的问题. 在开发的间隙或多或少的进行了部分修改

总结: 功能虽然是按时完成了, 但未能推动进行项目革新, 无法简化当前的开发模式与技术的换代(历史代码繁多).

2017/06

产出

  • notify-ear/ejb/web 消息通知重构
  • 重新梳理缓存的启动方式
  • 类路径下的jpa multi entity jar支持

原消息通知中的功能与页面停留在ALE阶段扩展封闭, 需要添加一种类型的通知较为困难. 重构后需要确保对新类型的通知扩展方便, 同时还要能介入该类型通知配置.

原有的通知类型有邮件/微信/邮件/站内/短信, 按新需求需要加入对客户端(App)的提醒(jpush). 在保留原功能的基础上添加新类型的提醒.

重构后支持:

  • 可自定义通知类型的消息通知(实现对应的类即可)
  • 沿用原来的配置项, 兼容配置
  • 对不同类型的通知使用不同的创建向导实现区别化对待
  • 消息通知的可监控性, 提醒哪些通知成功与失败
  • 方便的Builder Api
  • 生产消费的隔离对

2017/07

进入年中, 公司的项目多了起来, 需要对不同的项目进行支持.

  • 项目支持(anli/doublestar/772)

主要支持问题:

  • 首页样式
  • 个人设置
  • 用户的扩展数据
  • 快捷菜单
  • 多模块间页面切换

在对项目的支持阶段发现修改的净是一些页面的问题, 通过这部分修改最终也为接下来的控制台需求提供了思路. 不光光在于新需求导致的修改, 渐渐发现在代码移交研发中心阶段的代码质量问题凸显.

2017/08

  • 项目支持
  • 对支持阶段发现的问题整理
  • 组件版本发布1.0.9
  • 应用版本发布1.1.0.beta

对遗留下来的不规范代码进行规范化, 首页部分的代码进行重构. 实现一套完整的多模块建页面切换方式(由于去掉了iframe整个应用等同于一个SPA, 多模块建的页面却换就等同于父组件与子组件之间的通信).

首页重构中预留可配置的项目, 首页常常应为不同的项目有定制选项. 为常变换的部分提供可配置点方便以后的修改.

2017/09

  • 项目支持
  • 支持阶段功能沉淀
    • 常用菜单/操作菜单
    • 国际化
    • 应用程序启动器优化
    • 前端过场提示
    • 登录超时提示
  • 菜单构建逻辑梳理

这部分时间主要是对项目的支持, 在项目支持之余就对支持阶段形成的固定功能进行整理最终合并入平台功能.

2017/10

  • 项目支持
  • 日志部分的功能
  • 微信配置功能
  • 优化系统初始加载的数据
  • 应用版本发布1.1.0

新增了日志部分的功能, 提供动态日志Appender(file/jms/jpa), 文件日志的导出.

微信的配置功能, 需要满足系统中支持多个公众号的需求. 在系统中存在一个默认使用的公众号. 应用的启动中需要加载一部分的数据库数据, 这部分一般涉及到大量的数据关联, 对其进行加载优化, 并为启动部分的数据提供reload功能.

国庆期间对开发机进行了系统重装, 国庆后将原系统中的资源组代码合并入平台中(妥协办法, 资源组部分的代码还是很混乱的). 根据原菜单逻辑开始对菜单部分的功能进行重构, 菜单部分存在多种问题, 数据混用首页菜单树构建代码混乱(10年前的代码?).

2017/11

  • 项目支持
  • 前端页面的component化
  • 重构菜单-资源-权限
  • component的freemarker方式

对控制台部分的前端组件进行组件化, 采用Ext.define + ViewModelAction方式来重构代码, 代码结构清晰化.

重构了菜单-资源-权限的代码与页面逻辑, 将此部分的功能集成在同一个页面中能快捷的做到新建与编辑与菜单-资源-权限建的关系了解.

菜单部分的重构, 重新考虑不同端的数据表现(不同的端根据其rootId来确认, 相当于给定一个菜单类型了表示其所属的端), 采用统一中数据结构来表示菜单.

其他优化:

  • 组件版本发布1.1.0
  • console 1.1.x 密码找回
  • 合并i18n应用入控制台
  • 全局系统参数
  • 文件部分功能
  • 可配置化的首页

2017/12

  • 文件功能的设置
  • 系统参数的设置方式考虑
  • 统一格式的Response
  • 全局异常的考虑(未进入开发阶段)
  • 组件版本发布1.1.1.RC1
  • console 1.2.0.RC1

文件设置需要考虑文件备份与还原, 在存储文件数据时候宜采用相对路径. 在考虑做系统层面的设置时候往往都采用存入系统参数的方式, 但是对于运维人员来说找寻系统参数去修改是件头痛的事情, 所以需要重新考虑如何对这部分系统参数的设置在前端页面部分做出修改. 采用功能页面即配置界面的方式, 来归并系统参数的配置. 在对应的功能界面即能找到对应的配置选项(日志的系统配置/文件的系统配置等等都适用这一原则).

在开发与运维/客户的交流中往往费力, 全局的系统异常力图解决这一问题. 全局的系统异常应该包括简要的异常描述, 异常编码, 异常的追溯码. 在不过多的暴露信息的情况下简要的描述给运维/客户看, 让其能大致了解异常的原因, 如果遇到了解决不了的问题则将异常的追溯码反馈给对应的开发或高级运维, 通过查询对应的异常日志(追溯码作用)则能定位到对应的异常日志.

项目总结

控制台与组件的人员无法保证, 始终只有1~2人参与所以其迭代速度还是比较慢的, 人员少还进一步使得缺少思维的迸发, 新的功能与想法也最终寥寥.

版本难以控制, 项目的需求对于平台功能是一个倒置的关系, 公司奉行项目为先一切与项目的需求为准, 在这样的条件下对于版本的控制就显得比较困难了.

缺少自动化, 限于公司对权限的管理严格想要实施一套自动化的DevOps的也就进一步加剧了.
